Visual Localization Tests

Purpose

The purpose of this test was to determine the accuracy of the primary localization system used by our robot.

Method

To determine efficiency of ArUco localisation testing was conducted independently of other localisation methods. 50x50cm ArUco tag was put at the origin of the world, facing y-axis. Robot facing the opposite of y-axis was put in 16,922 different positions along lines x=[-3, -2.5, -2, -1.5, -1, -0.5, 0, 0.5, 1, 1.5, 2, 2.5, 3] (all values are in meters). At each position, the distance between actual position and the estimated position (error) was measured. The resulting error heatmap is presented below.

Result Interpretation

Not surprisingly, the closer robot is to the marker, the more accurate is its estimate. This has a crucial importance for determining the effective range for marker detection. Technically, we could detect the marker from a big distance (e.g. 11 meters), but this test tells us that estimate based on such marker is likely to be inaccurate.

Moreover, position estimates tend to be more accurate when robot sees the marker at an angle rather than when the marker is facing the camera.

This knowledge resulted in a improved design of the robot. We maximised the ArUco detection range by rotating fron cameras of the robot by 30 degres. At the same time, we limitted the detection range for the rear camera to guarantee that the estimates before entering the blindzone are accurate.

Hardware-Backend Integration Tests

Purpose

Main purpose of the hardware-backend integration tests is to determine whether the robot's controller is able to correctly manage various states of the of the robot, execute (or not) commands in given states, fetch tasks from the server, and update the database with the current state of the task. It is especiallly important as many subsystems have to properly cooperate in order to carry out specifc task. What is more, robot states are more fine grained than state of the tasks that are store in the server, so the synchronisation between the robot and the server has to occur in specific order and at specific phases of command execution.

Method

Hardware modules integration has been evaluated by executing different scenarios and measuring in how many trials the system behaves as expected. The following scenarios have been tested:

  1. Robot shuts down after it receives Abort Safety command in Idle state
  2. Robot shuts down after it receives Abort LowBattery command in Idle state
  3. Robot executes ZigZag Task
  4. Robot does not execute issued Circular Task while executing ZigZag Task
  5. Robot does not shut down when it receives Abort LowBattery Command during cleaning task
  6. Robot shuts down when when it receives Abort Safety Command during cleaning task
  7. Robot performs Circular task that is scheduled in the future at the appropriate time
Results and Interpretation

We have executed each scenario 5 times with a 100% success rate. The system behaving as expected in each test case gave us confidence that the subsystems has been integrated correctly.

Measuring Pattern Execution Time

Measuring the actual time of the program execution turned out to be a difficult task. We had only one machine capable of running the simulation and in that machine the simulation rate oscillated around 0.05-0.10x the actual time. Meaning that it took from 10 to 20 seconds for 1 second to pass in the simulation. For that reason we had to make some "smart" estimates regarding the execution times of our patterns, as it was infeasible to carry out multiple trials using full setup.

We decided to measure how much time it takes to move the robot from point A on the one side of the court to point B on the other side of the court in a straight line fashion. We compared the results for two cases:

  1. the robot blindly moved from A to B
  2. the robot used our navigation module that utilises odometry and ArUco navigation

Not surprisingly, the second case took longer than the first one. On average by 19%. There are two main reasons for that. Firstly, when using ArUco navigation there is some "jerky" motion caused by the inaccuarate pose estimation making the robot adjust it's pose. Secondly, due to drag and direction estimation inaccuracy. Robot does not move in a straight line, rather than a curve.

Knowning that, we run the simulation without using the ArUco navigation which speeded up the simulation to bearable time. In such circumstances, we achieved following results:

Pattern Type Repetitions (Min - Max) Running Time w/o. ArUco localization Estimated (Min - Max) Running Time w. ArUco localization
Zig-Zag 10 9m 28s - 10m 6s 11m 16s - 12m 1s
Circular 10 23m 13s - 23m 41s 27m 38 - 28m 11s

We find the estimated times to be within acceptable range. Zig-Zag pattern which lasts almost 12 minutes falls little behind the time it takes for an average person to sweep the court (from our research it is around 5 minutes per half). For the circular pattern this time is over 2 times longer, but at the same time circular traverses the same leg two times. However, due to its duration, we don't expect people using it midgame. It would rather act as a start/end of day court maintenance program. It is important to stress the fact that the exact time results heavily depends on the configuration of the cleaning patterns. As the choice of parameters for pattern generation might differ between users.

Despite Claynce being slower than human player, these 10 minutes are freed for players / court maintainers. In this time they can warm-up, stretch, rest, talk with the partner, or clean up different parts of the court. Additionally.

Accessibility and Usability Tests

Testing Overview

Our team truly believes in giving each user a rich experience to control their Claynce robot with greater ease. Accessibility and usability continues to remain as an important aspect of our project, otherwise the user would ultimately not would want to use our product. Therefore, our team has created an accessibility and usability questionnaire to further understand our users better. As our team’s Participant Information Sheets and consent forms met the SDP Ethics Committee’s standards, our team has proceeded to collect 30 responses from SDP students and friends. Our team believes in the integrity of input, and these individuals are useful sources, because every individual plays a key role in helping our team continuously improve our user interface.
